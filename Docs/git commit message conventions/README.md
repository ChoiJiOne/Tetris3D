# git commit message conventions

이 문서는 git 커밋 메시지 컨벤션에 대한 문서입니다.
<br><br>


## 동기

이전의 개인 혹은 팀 프로젝트를 진행할 때는 git commit 메시지를 해당 작업에 맞게 적절하게 작성하였습니다. 하지만, 프로젝트를 진행하다보면 다양한 작업을 수행하게 됩니다. 예를 들면 코드를 작성할 수도 있고, 작성한 코드가 올바르게 동작하는지 확인할 수도 있고, 문서를 작성할 수도 있고, 스크립트를 작성할 수도 있고, 프로젝트 세팅을 진행할 수도 있습니다. 이처럼 프로젝트를 진행하면 다양한 작업들이 있는데, 이들을 *git log*에서 구분할 수 있도록 하기 위해서 git commit message 컨벤션을 정하게 되었습니다.
<br><br>


## git commit message conventions

이 프로젝트에서의 git commit message conventions은 다음과 같습니다.
- 코드 구현의 경우 앞에 `[DEV]`를 붙입니다.
  - ex. `[DEV] 셰이더 로딩 기능 추가`
- 버그가 발생한 경우 앞에 `[BUG]`를 붙입니다.
  - ex. `[BUG] 큐브에 라이팅 효과 적용 시 버그 발생`
- 버그를 수정한 경우 앞에 `[BUGFIX]`를 붙입니다.
  - ex. `[BUGFIX] 텍스트 렌더링 시 문자자 위 아래로 뒤집히는 버그 수정`
- 스크립트 작성의 경우 앞에 `[SCRIPT]`를 붙입니다.
  - ex. `[SCIPT] Visual Studio 솔루션을 생성하는 배치 스크립트 추가`
- 문서 작성의 경우 앞에 `[DOCS]`를 붙입니다.
  - ex. `[DOCS] 프로젝트 제안서 문서 추가`
- 코드 테스트를 수행한 경우 앞에 `[TEST]`를 붙입니다.
  - ex. `[TEST] 큐브의 충돌 처리 기능 테스트 코드 추가`
- 프로젝트 세팅을 수행한 경우 앞에 `[SETTING]`을 붙입니다.
  - ex. `[SETTING] Json 파일을 파싱하고 로딩하는 외부 라이브러리 추가`
- 다른 브랜치와 병합했을 경우 앞에 `[MERGE]`를 붙입니다.
  - ex. `[MERGE] 테스트 브랜치와 병합`
- 위의 정의된 경우로는 설명할 수 없을 경우 앞에 `[MISC]`를 붙입니다.
